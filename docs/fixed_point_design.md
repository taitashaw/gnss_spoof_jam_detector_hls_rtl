# Fixed-Point Design

Author: John Bagshaw — License: MIT (c) 2026 John Bagshaw

No floating point appears in any synthesizable path. Float is used only in the Python
golden/generator and in the test-bench accuracy comparison, never in the kernel body.
The detector is the own FFT (`fft_fixed.hpp`) plus the ddMap correlation and SQM
read-out (`ddmap_sqm_hls.cpp`). Widths below are read directly from those files.

## FFT data types

| Quantity | Type | Rationale |
|---|---|---|
| FFT data `fdata_t` | `ap_fixed<24, 4, AP_RND_CONV, AP_SAT>` | 4 int bits headroom, 20 frac, convergent rounding, saturate |
| Twiddle `twid_t` | `ap_fixed<18, 2, AP_RND_CONV, AP_SAT>` | \|W\| ≤ 1, 16 frac bits |
| Butterfly product `fmul_t` | `ap_fixed<34, 6>` (truncate+wrap) | the complex multiply before rounding back to `fdata_t`; plain trunc/wrap keeps the hot path short (no saturate/round logic), numerically negligible at 28 frac |

Scaling is the classic per-stage divide-by-2 (total /N), so values stay in range;
the output is ~ DFT/N. The /N factor and any uniform rescale are harmless for
acquisition: the correlation peak location and the early/late distortion ratio are
scale-invariant. Rounding is `AP_RND_CONV` (convergent) on the `fdata_t` results to
remove bias. The twiddle ROM `W[k] = exp(-j 2 pi k / N)` is generated by
`scripts/gen_fft_twiddle.py` into compile-time double literals that the compiler
converts to fixed point — a ROM, no runtime float. Full design and the numpy
accuracy result (≥ 50 dB SNR, 13–15 ENOB on signal cases) are in
`docs/fft_fixed_design.md`.

## ddMap correlation widths (`ddmap_sqm_hls.cpp`)

| Quantity | Type | Rationale |
|---|---|---|
| input I/Q normalize | `int16 >> 16` -> `fdata_t` (~[-0.5, 0.5)) | a shift, not a divide |
| spectral-product rescale | `2^14` left shift, saturate to `fdata_t` | `PROD_SHIFT = 14`; brings the small triple-transform product back to O(1) for the IFFT (scale-invariant) |
| coherent accumulator `acc_t` | `ap_fixed<24, 6, AP_RND_CONV>` | corr is `fdata_t`; sum over `N_BLK = 4` -> needs ~6 int bits; narrow so the power square is fast |
| saturating product `psat_t` | `ap_fixed<34, 6, AP_RND_CONV, AP_SAT>` | the `blk_fd * conj(code_fd)` products, saturating |
| power `pw_t` | `ap_ufixed<48, 14>` | `re*re + im*im`; narrow enough that the squarings and the peak compare close 2.5 ns |

The carrier wipeoff is done by the host before the stream enters; the kernel applies
the `2^14` gain because the two scaled (/N) FFTs and the spectral multiply shrink the
correlation to ~1e-5, which would otherwise sit at the `fdata_t` resolution floor.
The gain is a left shift (a power of two), and being uniform it does not move the
peak or change the distortion ratio.

## SQM read-out

The `|corr|^2` profile is reduced to a peak with a **4-lane partial-max** (element
`i` -> lane `i % 4`) so the max-update recurrence has distance 4 and pipelines at the
target clock; the 4 lane-maxima are combined after the loop. The early and late
samples are taken at `± SAMP_PER_CHIP/2` (±0.5 chip), and the spoof statistic is
`distortion = |E - L| / (E + L)` in Q16. The power outputs to `s_axilite` are scaled
to integers by a pure left shift (`pw_to_u48`, `<< 20`) — relative magnitudes only,
since they are compared as ratios.

## How float is avoided in hardware

The FFT twiddles are a constant ROM (no `cos`/`sin` at run time); every scaling and
the input normalization and the product gain are power-of-two shifts; the only
genuine division (`|E - L| / (E + L)` for the distortion) is one sequential divide
outside the hot loop, on narrowed operands. The numpy reference uses float; the
fixed-point kernel matches it to ≥ 50 dB FFT SNR and reproduces the golden detector
peak/distortion (0.3305 vs 0.3305; peak phase exact).

## Legacy streaming front-end (superseded)

The earlier metric engine's width budget (NCO Q14 LUT, per-window power/correlation/
Doppler accumulators, division-free log-domain C/N0, saturated 0..255 score features)
described the **legacy** streaming detector and is **not** part of the current ddMap/
SQM design. Those constants live in the legacy section of `hls/include/gnss_config.hpp`;
see README §11.
