# HLS Synthesis Report

Author: John Bagshaw — License: MIT (c) 2026 John Bagshaw

Every number on this page is copied verbatim from the Vitis HLS tool's own
report. Nothing here is estimated by hand or invented. The raw reports are
committed under `docs/synth/` and are regenerated by `make hls`.

## Provenance

- Tool: Vitis HLS 2025.2 (Build 6295257, Nov 14 2025).
- Command: `vitis-run --mode hls --tcl hls/vitis_hls/run_hls.tcl`
  (this install exposes the HLS flow through `vitis-run`/`v++`; `make hls`
  detects and uses it automatically).
- Top function: `gnss_metric_hls`.
- Target device: `xczu7ev-ffvc1156-2-e` (Zynq UltraScale+, family zynquplus).
- Clock target: 5.00 ns (200 MHz).
- Flow run: C simulation, C synthesis, and RTL IP export — all completed with
  return code 0.

| Clock | Target | Estimated | Uncertainty |
|---|---|---|---|
| ap_clk | 5.00 ns | 3.625 ns | 1.35 ns |

The estimated critical path (3.625 ns) is below the 5.00 ns target less the

## Throughput — accumulation loop (from `gnss_metric_hls_Pipeline_ACC_LOOP_csynth.rpt`)

| Loop | Iteration latency | Initiation Interval (achieved) | II (target) | Pipelined |
|---|---|---|---|---|
| ACC_LOOP | 4 | 1 | 1 | yes |

The per-sample accumulation loop achieves **II = 1**: one input sample is consumed
every clock.

### Latency is not bounded by synthesis

The csynth report lists the loop trip count and the absolute latency (cycles) as
`?`. This is correct and expected: `WINDOW_SIZE` is a run-time input, the loop runs
until `tlast` rather than a compile-time constant, so the total cycle latency is
unbounded at synthesis. Synthesis therefore reports throughput (II = 1) and the
per-iteration latency (4 cycles), but **not** a single absolute latency figure, and
none is invented here.

The real per-window latency is measured directly in XSim by
`axis_latency_counter` (first accepted input beat to the metrics output beat,
including stalls): **1021 cycles with no backpressure and up to 2038 cycles under
burst backpressure** at the default 1024-sample window. That measured number, not a
synthesized one, is the latency this design reports.

## Utilization estimates (from `gnss_metric_hls_csynth.rpt`)

| Resource | Used | Available | Utilization |
|---|---|---|---|
| BRAM_18K | 0 | 624 | 0% |
| DSP | 4 | 1728 | ~0% |
| FF | 1735 | 460800 | ~0% |
| LUT | 4026 | 230400 | 1% |
| URAM | 0 | 96 | 0% |

The metric engine is tiny relative to the ZCU104-class part: four DSPs (the
correlation and cross-product multipliers), about 1.7k flip-flops and 4k LUTs,
and no block RAM. There is ample room for replication (multiple PRN channels or
correlator banks) within one device.

## Notes

- These are post-C-synthesis estimates from Vitis HLS, not post-implementation
  results. Logic synthesis to final utilization and timing closure happen in
  Vivado place-and-route, listed as a deferred phase in the README roadmap.
- The exported RTL IP (`xilinx_com_hls_gnss_metric_hls_1_0.zip`) is produced by
  the same run and is the drop-in replacement for the behavioral
  `gnss_metric_hls_model.sv` via the one-line switch in
  `vivado/compile_order.tcl`.
